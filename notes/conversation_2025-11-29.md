## 현재 GitHub Copilot의 구조

### ✅ 지원하는 것
1. **단일 글로벌 지침서** (`.github/copilot-instructions.md`)
   - 프로젝트 전체에 적용되는 하나의 지침
   - 모든 파일과 컨텍스트에서 동일하게 작동

2. **작업공간별 설정**
   ```json
   // .vscode/settings.json
   {
     "github.copilot.enable": {
       "*": true,
       "yaml": false,
       "plaintext": false
     }
   }
   ```

3. **컨텍스트 인식**
   - 열려 있는 파일들을 자동으로 참고
   - 프로젝트 구조 이해
   - 언어/프레임워크 자동 감지

### ❌ 지원하지 않는 것
- 파일/폴더별 독립적인 지침서
- 작업 유형별(프론트엔드/백엔드) 에이전트 분리
- 조건부 지침 활성화/비활성화
- 여러 개의 instruction 파일

### 방법 2: 파일별 주석 활용
```typescript
// components/events/EventCard.tsx
/**
 * @copilot
 * 이 컴포넌트는 공연 카드입니다.
 * - 모바일 퍼스트 디자인
 * - Tailwind만 사용
 * - 서버 컴포넌트 우선
 */
```

### 방법 3: Claude Code (Cline) 병행 사용
복잡한 작업이나 세부 에이전트가 필요한 경우:
1. **GitHub Copilot**: 일상적인 코드 자동완성
2. **Claude Code**: 복잡한 리팩토링, 아키텍처 변경, 서브 에이전트 필요한 작업

### 방법 4: 팀별 지침서 (문서로만)
```
docs/
  ├── guides/
  │   ├── FRONTEND_GUIDE.md
  │   ├── BACKEND_GUIDE.md
  │   ├── DATABASE_GUIDE.md
  │   └── TESTING_GUIDE.md
```
Copilot에게 직접 적용되진 않지만, Copilot Chat에서 참조할 수 있습니다:
```
@workspace /explain 이 컴포넌트를 FRONTEND_GUIDE.md 규칙에 따라 리팩토링해줘
```

## 🤔 추가 질문

### 질문 A: 이벤트허브 vs 공연 상세
제가 이해한 게 맞는지 확인하고 싶어요:

- **공연목록**에서 공연 클릭 → **특정 이벤트 페이지** (`/events/[id]`)
- **이벤트허브 메인**에서 공연 클릭 → **특정 이벤트 페이지** (같은 페이지)

즉, "이벤트 허브"는:
- **메인 페이지**: 여러 공연의 이벤트허브를 모아서 보는 곳
- **특정 페이지**: 개별 공연의 상세 정보 + 커뮤니티 기능

이렇게 이해하면 될까요?

그렇다면 라우팅은:
```
/events          - 공연 목록 페이지
/events/[id]     - 특정 이벤트 페이지 (= 이벤트허브 특정 페이지)
/events/hub      - 이벤트허브 메인 페이지 (여러 공연 한눈에)
```

맞나요?

### 질문 B: 공연목록 vs 이벤트허브 메인의 차이

둘 다 여러 공연을 보여주는데, 차이점은:

**공연목록** (`/events`):
- 모든 공연 (과거/현재/미래)
- 필터/정렬 중심
- 정보성

**이벤트허브 메인** (`/events/hub`):
- 진행 중/임박한 공연 중심
- 실시간 활동(제보, 양도) 강조
- 커뮤니티성

이렇게 구분되는 건가요?

### 질문 C: 타임테이블 표시 방식

특정 이벤트 페이지의 타임테이블 섹션:

**옵션 1 - 간략 버전**
```

18:00 게이트 오픈
19:00 오프닝 아티스트
20:00 메인 아티스트
[전체 타임테이블 보기 →]
```

**옵션 2 - 전체 표시**
```
⏰ 타임테이블
┌──────────────────┐
│ 18:00 게이트 오픈│
│ 18:30 입장 완료  │
│ 19:00 오프닝     │
│ 19:40 인터미션   │
│ 20:00 메인       │
```

**옵션 3 - 인터랙티브**
```

☑ 20:00 메인 아티스트
```

어떤 방식이 좋을까요?

### 질문 D: 아티스트 정보의 중복

특정 이벤트 페이지에 아티스트 전체 정보(호응법, 셋리스트)가 나오면:

1. **아티스트가 3명 이상**일 경우 페이지가 너무 길어질 수 있는데 괜찮나요?
   - 예: 페스티벌 라인업 10팀 → 각 팀마다 호응법 + 셋리스트

2. **아티스트 카드 형태로 접기/펼치기**를 추가할까요?
```

[아티스트 1 ▼]  ← 클릭 시 펼쳐짐

  - 셋리스트

[아티스트 2 ▶]  ← 접힌 상태

[아티스트 3 ▶]
```

3. 아니면 **요약만 보여주고 [아티스트 상세 보기]** 버튼으로 아티스트허브로 이동?

### 질문 E: 현장 제보 우선순위

- **최신순** (가장 최근 제보 상위)
- **신뢰도순** (윌슨 스코어 높은 것 우선)
- **타입별 탭** (줄 상황 | MD 재고 | 시설 | 긴급)
- 혼합?
